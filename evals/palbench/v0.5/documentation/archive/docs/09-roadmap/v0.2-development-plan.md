# AgentPal v0.2 Development Plan

AgentPal v0.2 moves the workspace from release-candidate hardening into product-path development. The goal is not to add a runtime, UI, installer, scanner, validator, or automated multi-agent engine. The goal is to make the no-code Pal layer easier to use, easier to verify, and easier to evolve.

## v0.2 Goal

v0.2 should make AgentPal feel like a practical AI team framework:

- users can ask PalSmith to create a Pal or AI team through a complete no-code path;
- users can start with Mira for ordinary work without manually choosing every Pal;
- official Pals have realistic examples that explain what good work looks like;
- Capability Inventory has a minimal usable manual profile shape;
- PalBench Light becomes a repeatable regression suite;
- Runtime Adapter guidance is stable across Codex, Claude Code, and generic CLI agents.

## What v0.2 Does Not Do

v0.2 does not add:

- native `/pal` execution API;
- desktop or web UI;
- CLI, installer, daemon, scanner, validator, or service;
- automatic Capability Inventory probing;
- active Deep Conductor execution;
- active Subagent Mode or automatic multi-runtime orchestration;
- hard-coded task/domain to Pal routing;
- PalSmith as an executable importer, exporter, scanner, or builder.

## Module 1: PalSmith Productization

Current basis:

- PalSmith is registered as the official no-code Pal asset governance Pal.
- Pal creation, AI team design, import/export, health checks, readiness review, and task-package templates already exist.

Missing:

- a user-copyable end-to-end path from plain goal to Pal/team design, file task package, self-test, repair package, and usage example;
- clearer examples that do not require users to understand the full Pal Pack structure first.

v0.2 target:

- make PalSmith the shortest safe path from "I need an AI teammate/team" to a draft Pal Pack or team design;
- keep all writes under explicit Runtime Task Packages and evidence review.

Acceptable artifacts:

- `docs/06-palsmith/palsmith-end-to-end-productization.md`;
- PalSmith examples for creating a first professional Pal and an AI team from a user goal;
- creation task-package templates and a PalSmith E2E self-test.

Boundary:

- PalSmith produces plans, questions, packages, and reviews. Host runtimes execute approved file work.

## Module 2: Mira-First Daily Use

Current basis:

- Mira is the default Main Pal, Leader Pal, and Conductor.
- Runtime Response Gate and deliverable-aware task judgement already define owner judgement, Fast Route, and staged Task Packages.

Missing:

- a user-level explanation of when to ask Mira, when to call `/pal Name`, and how Mira handles composite deliverables.

v0.2 target:

- make Mira-first the default everyday path for users who do not want to pre-select Pals;
- preserve owner Pal responsibility and avoid making Mira a substitute specialist.

Acceptable artifacts:

- `docs/02-getting-started/mira-first-usage.md`;
- Mira examples for daily tasks, composite deliverables, and owner judgement;
- a Mira-first usage self-test.

Boundary:

- Mira coordinates, routes, prepares Task Packages, and summarizes. Specialist Pals still own specialist bodies.

## Module 3: Official Pal Real Task Example Library

Current basis:

- 9 official Pals are registered and have role assets, output contracts, skills, workflows, runbooks, and evals.

Missing:

- a consistent example pattern showing user request, Pal judgement, context needs, output contract, good response, bad response, and verification.

v0.2 target:

- create an example library plan with at least 3 real task example plans per official Pal;
- land the first concrete samples for Mira, PalSmith, Atlas, Vega, and Quinn.

Acceptable artifacts:

- `docs/07-official-pals/official-pal-example-library-plan.md`;
- first sample files under relevant Pal example directories.

Boundary:

- examples are non-binding learning material, not route maps.

## Module 4: Capability Inventory Minimal Usable Profiles

Current basis:

- Capability Inventory methodology, runtime-profile examples, and profile templates exist.

Missing:

- a minimal usable design that tells users how to fill profiles manually and how the profiles feed AI judgement without becoming routing rules.

v0.2 target:

- define the smallest profile fields that are useful for runtime, model, and Skill judgement;
- keep all examples illustrative and non-binding.

Acceptable artifacts:

- `docs/05-orchestration-methodology/capability-inventory-minimal-usable-design.md`;
- refreshed README guidance for runtime, model, and Skill profile folders;
- existing JSON examples continue to parse and state they are illustrative.

Boundary:

- no automatic scanner, validator, or environment probe.

## Module 5: PalBench Light Suite

Current basis:

- R33 provides a six-sample smoke validation and a future PalBench design note.

Missing:

- a repeatable 20 to 50 item light suite structure with named sample types and scoring notes.

v0.2 target:

- create a first 10-sample PalBench Light set that can grow to 20 to 50 cases;
- cover ordinary routing, direct Pal calls, composite deliverables, context slicing, false completion, no-code boundary, release review, PalSmith creation, AI team creation, and runtime adapter binding.

Acceptable artifacts:

- `evals/palbench/v0.5/documentation/archive/docs/research/palbench-light-suite-plan.md`;
- `evals/palbench-light/README.md`;
- `evals/palbench-light/task-index.md`.

Boundary:

- PalBench Light is a regression suite for AgentPal behavior, not a statistical model benchmark.

## Module 6: Runtime Adapter Stability Regression

Current basis:

- Codex, Claude Code, and generic CLI usage guides exist.
- Thin binding templates, examples, and evals exist.

Missing:

- one stability plan and one regression suite that cover workspace mode, project-first install, existing binding upgrade, remove/reinstall, Pal list refresh, core gate propagation, project-root separation, and failed install troubleshooting.

v0.2 target:

- make adapter behavior easier to test after any core gate or binding template change;
- reduce project-root confusion in real external-project use.

Acceptable artifacts:

- `docs/04-runtime-guides/runtime-adapter-stability-plan.md`;
- `evals/runtime-adapters/runtime-adapter-regression-suite.md`.

Boundary:

- prompts and docs remain thin bootstraps. They must point back to AgentPal core gates rather than copying full rules.

## v0.2 And v0.3 Boundary

v0.2:

- productized PalSmith and Mira-first paths;
- realistic official Pal examples;
- minimal manual Capability Inventory profiles;
- PalBench Light regression suite;
- runtime adapter stability checks.

v0.3:

- Deep Conductor prototype exploration;
- multi-Pal collaboration simulations that may later become runtime prototypes;
- Routing Reward Memory writeback experiments;
- stronger Capability Inventory capture if a separate runtime product line is approved.

## Acceptance

v0.2 planning is acceptable when the roadmap, task pool, first PalSmith and Mira examples, official Pal example plan, Capability Inventory design, PalBench Light plan, and Runtime Adapter stability plan all exist and preserve the no-code Pal layer boundary.
