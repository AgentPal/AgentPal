# v0.3 Public Capability Summary

This summary describes the public-safe capability boundary for AgentPal v0.3 release-candidate preparation.

## Current v0.3 Capability

AgentPal v0.3 is a no-code orchestration prototype line for host Agent Runtimes. It provides Markdown / JSON protocols, templates, examples, and evals that help a host Runtime coordinate complex work with Pal identity, bounded context, task packages, verification, synthesis, and memory writeback candidates.

## What Is Ready For Release-Candidate Preparation

- Deep Conductor no-code master loop for complex and project-level goals.
- Project Conductor no-code task maps and next-round Runtime Task Packages.
- Context Packet and Context Access List boundaries.
- `/pal Name` and `@Pal` collaboration protocols as plain-text behavior.
- Owner + Verifier evidence workflow.
- Parallel Independent Review isolated reviewer workflow.
- Plan -> Execute -> Verify staged package.
- Runtime-installed Skill Orchestration as candidate, availability, fallback, verification, and usage-memory package logic.
- Cross-Runtime Pal Memory through portable no-code snapshots and summaries.
- Context Budget and Context Usage Reports as qualitative planning and reporting artifacts.
- PalBench Collaboration qualitative regression coverage.

## Host Runtime Boundary

The host Runtime still performs file reads, writes, commands, rendering, browsing, publishing, tool calls, Skill use, plugin use, and MCP calls. AgentPal may prepare task packages for these actions, but completion claims require host Runtime evidence.

## Runtime Skill Boundary

Runtime-installed Skills, plugins, MCP tools, browser tools, document tools, repository tools, and similar capabilities are host Runtime candidates. AgentPal does not own, install, scan, or execute them. Availability must be checked in the current Runtime before use.

## Subagent And External Agent Boundary

AgentPal can prepare no-code handoff packages for a host Runtime or user. AgentPal does not start Subagents, call external Agents, manage child workflows, or run multi-agent execution. If the current host Runtime does not prove support and permission, the correct status is `unavailable`, `unknown`, `blocked`, or `not-run`.

## Token And Cost Boundary

Context Budget is qualitative. It records read tiers, memory reuse, missing context, verification context, and tradeoffs. It is not exact token metering or cost calculation unless a host Runtime provides separate exact metering evidence.

## PalBench Boundary

PalBench Collaboration is a qualitative regression suite for no-code collaboration behavior. It is not a statistical benchmark and does not prove foundation-model performance.

## Public Claim Rule

Public v0.3 materials should say "ready to enter release preparation" when the checklist passes. They should not say a tag, push, GitHub Release, automatic execution, active Subagent Mode, external Agent calls, exact token metering, or installed Runtime Skill availability exists unless there is current evidence.
