# AgentPal v0.5 Remote Operation Authorization Guide

Status: R212 local authorization guide.

R212 performs no remote operation.

Preparing a release package is not authorization to publish it.

## Operations That Require Explicit User Authorization

The user must explicitly authorize each remote operation:

- push branch;
- create tag;
- push tag;
- create GitHub Release;
- upload release assets;
- publish to any package registry.

Authorization should name the exact branch, tag, release title, release notes
body, and asset path where applicable.

## Before Push Branch

Preflight:

- `git status --short --branch`
- confirm current branch
- confirm expected ahead count
- confirm no unreviewed local changes
- confirm no prohibited files in the commit range

If push is rejected, stop. Do not pull, fetch, rebase, or merge without a new
explicit user instruction.

## Before Create Tag

Preflight:

- confirm target commit hash
- confirm tag name
- confirm tag does not already exist locally
- confirm user explicitly authorized tag creation

Correction path:

- If tag creation fails, stop and report.
- Do not delete or overwrite tags without new explicit authorization.

## Before Push Tag

Preflight:

- confirm local tag exists and points to the intended commit
- confirm tag name exactly matches user authorization
- confirm push tag is separately authorized

Correction path:

- If tag push fails, stop and report.
- Do not pull/fetch/rebase/merge to fix tag push without authorization.

## Before Create GitHub Release

Preflight:

- confirm tag exists remotely
- confirm release title
- confirm pre-release vs stable release status
- confirm release notes body path
- confirm asset path if uploading assets
- confirm user explicitly authorized GitHub Release creation

Correction path:

- If release creation fails, stop and report the failure point.
- Do not retry with a different tag, title, or release status without
  authorization.

## Before Upload Release Assets

Preflight:

- confirm asset path exists
- confirm checksum or size if required
- confirm asset name
- confirm user explicitly authorized upload

Correction path:

- If upload fails, stop and report.
- Do not replace, delete, or rename remote assets without new authorization.

## What R212 Does Not Do

R212 does not:

- push;
- pull;
- fetch;
- create a tag;
- push a tag;
- create a GitHub Release;
- upload release assets;
- publish to npm or any package registry.

## Required Release Boundary

Any future remote release report must clearly state which operations were
actually executed and which remain `not-run`.
