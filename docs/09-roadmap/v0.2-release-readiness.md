# AgentPal v0.2 Release Readiness

This document assesses whether the v0.2 first-phase work is ready to enter `v0.2.0-rc.1` release preparation.

It is a readiness assessment for the no-code Pal layer. It does not make AgentPal an Agent Runtime, a multi-agent runtime, an execution layer, a CLI, an installer, a scanner, a service, or a UI.

## First-Phase Goal

The v0.2 first phase set out to make AgentPal easier to use, easier to explain, and easier to review:

- PalSmith can guide Pal and AI team creation through copyable Runtime Task Packages.
- Mira-first usage gives users one default entry point when they do not know which Pal to call.
- Official Pal examples show realistic tasks for all bundled Pals.
- Capability Inventory gives manual profile inputs for AI judgement.
- PalBench Light gives a qualitative release regression suite.
- Runtime Adapter guidance makes thin project binding easier to install, verify, remove, and troubleshoot.

## Current Public Claims That Are Supported

AgentPal v0.2 can claim:

- it strengthens the PalSmith no-code Pal and AI team creation path;
- it strengthens Mira-first daily use;
- it adds realistic official Pal task examples;
- it adds minimal manual Capability Inventory profiles as AI judgement inputs;
- it adds PalBench Light qualitative release regression;
- it strengthens Runtime Adapter thin binding guidance and troubleshooting;
- it remains a Markdown / JSON / protocol workspace for existing runtimes.

## Claims That Are Not Supported

AgentPal v0.2 must not claim:

- automatic Deep Conductor execution;
- autonomous multi-agent runtime behavior;
- automatic detection of all runtimes, models, plugins, MCP tools, or Skills;
- statistically significant performance proof;
- replacement of Codex, Claude Code, OpenHands, or any other runtime;
- PalSmith as an executable importer, exporter, scanner, or builder.

## v0.2 And v0.3 Boundary

v0.2 is the first integrated product path for the no-code Pal layer:

- productized PalSmith creation flows;
- Mira-first entry and staged Task Package behavior;
- official Pal example library;
- manual Capability Inventory profiles;
- PalBench Light regression;
- Runtime Adapter thin binding stability guidance.

v0.3 may explore deeper orchestration, richer team lifecycle work, stronger runtime-hosted evidence capture, and more advanced capability profile maintenance. Those are not current v0.2 behavior.

## Release Readiness Recommendation

Recommendation: enter `v0.2.0-rc.1` release preparation as a candidate.

Reasoning:

- all six first-phase modules have public artifacts;
- integration evidence shows the modules point to each other through README, docs index, RESOURCE_INDEX, examples, and evals;
- current boundaries are preserved: no runtime code, no static routing, no future design presented as current behavior;
- remaining work is release preparation and manual host replay, not a blocker for starting rc preparation.

## Release Blockers

No blocker is identified for entering `v0.2.0-rc.1` release preparation.

Known pre-release risks:

- Claude Code and Generic CLI host replay should still be run manually before claiming broad runtime-host validation.
- PalBench Light has a dry-run subset and clear scoring rules, but it remains qualitative release regression.
- Capability Inventory examples are illustrative and should not be presented as the user's current environment.

## Minimum Checklist Before `v0.2.0-rc.1`

- Run the JSON parse check for workspace JSON files.
- Run the no-runtime artifact check on release changes.
- Run static-routing phrase checks and review any negative-example hits.
- Review future-boundary language.
- Scan public release files for internal local paths and private data.
- Run at least the v0.2 integration matrix.
- Run at least one PalBench Light dry-run sample from each major area when time allows.
- Manually verify one Claude Code thin binding path and one generic CLI thin binding path, or record them as `not-run` with a clear blocker.
- Confirm README, docs README, and RESOURCE_INDEX link the v0.2 readiness, capability summary, and integration matrix.
- Create the release notes only after the above checks have evidence.
