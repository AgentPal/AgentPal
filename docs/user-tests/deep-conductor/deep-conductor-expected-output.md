# DeepConductor Expected Output

The exact words can vary. The structure should be visible.

## Required Structure

```text
Task Intake
Team First Discovery
Routing Decision
Rejected Assignment Reasons
Execution Graph
Pal Work Plans
Asset Preflight
Execution Trace
Owner Assignment Integrity Gate
Closure Gate
Quinn Final Verification
Final Deliverable
```

## What Good Output Looks Like

### Task Intake

The answer identifies the request as a multi-role test planning and user-facing delivery task.

### Team First Discovery

The answer checks existing Team Packs before creating a new team. For AgentPal v0.6 tester recruitment, `marketing-growth-team` should usually be considered and reused if it fits.

### Routing Decision

The answer explains selected owners and rejects unfitting owners.

Expected common roles:

- Mira: coordination and DeepConductor trace.
- Marketing Growth Team: first-user recruitment and testing communication.
- Nova: target users and test scope.
- Vega: channel assumptions and feedback-question design.
- Harper: user-facing copy.
- Rhea: no-code / runtime / release boundary.
- Quinn: final verification.

### Pal Work Plans

Each selected Pal should say what it will use:

- persona / thinking style;
- knowledge;
- Skill;
- Workflow;
- Memory scope;
- Agent or model configuration, if any.

### Asset Preflight

The answer should connect work to the selected Team Pack and selected Pal assets. Tools or host runtime alone are not Pal assets.

### Execution Trace

Each material output should map back to an owner.

### Owner Assignment Integrity

The answer should prove that selected participants actually participated, or give a legal skip / block reason.

### Closure Gate

The answer should check all required steps before claiming done.

### Quinn Final Verification

Quinn should give a clear final verdict.

### Final Deliverable

The final output should be usable by the user, or clearly explain why it is not ready.
